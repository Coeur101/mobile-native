# AI-Native 移动端产品方案（安卓 App）

## 一、方案解读

这份方案的核心目标很明确：做一个以 AI 生成为中心的安卓端产品，让用户通过对话直接生成网页项目，在手机内完成查看、编辑、预览和管理。它不是传统的“聊天应用”，而是一个“AI 驱动的轻量网页工作台”，重点在于把“生成结果”沉淀成项目资产，而不是只保留对话记录。

当前方案已经具备一个较好的雏形，覆盖了产品目标、基础技术栈、页面结构、数据库设计和后续演进方向。但从落地角度看，还存在几个明显问题：一是文档结构不完整，`LLM 返回 JSON` 的代码块未闭合，导致后续章节全部混入同一段；二是业务边界还不够清晰，例如“用户配置”与“项目配置”需要拆分，API Key 是否允许用户自带也需要安全设计；三是工程落地层面缺少 MVP 范围、接口边界、权限模型、同步策略、错误处理和发布路径，容易在真正实施时反复返工。

所以，这份优化版方案的目标不是推翻原方案，而是在保留原方向的前提下，把它补成一份更适合执行、评审和拆任务的产品技术文档。

---

## 二、产品定位与目标

### 1. 产品定位

面向个人开发者、独立开发者、AI 爱好者和轻量网页需求用户，提供一个“在移动端通过自然语言生成网页项目”的 AI-native 安卓应用。

### 2. 核心价值

- 用对话代替手工搭建页面。
- 用项目代替一次性回答结果。
- 用移动端降低创作门槛，支持随时生成、修改和预览。

### 3. 产品目标

产品需要支持以下核心能力：

1. 用户通过 LLM 对话生成网页项目。
2. 用户可管理已生成项目，支持查看、新建、重命名、删除和继续编辑。
3. 用户可以在 App 内直接预览生成页面。
4. 用户的项目、配置、历史记录与账号绑定。
5. 支持系统默认模型，也支持用户配置自定义模型接入。
6. 最终以安卓 App 形态交付，优先保证 MVP 可快速上线。

### 4. MVP 范围

第一阶段建议只做最小可用闭环，避免方案过大：

- 登录：先做邮箱登录入口和微信登录入口，但都只落 mock 流程，不接真实认证。
- 生成：支持生成单页网页项目，但生成能力先由本地结构化 mock generator 提供。
- 管理：支持项目列表、详情、编辑、删除。
- 预览：支持 WebView 预览 HTML/CSS/JS。
- 存储：当前以本地缓存为主，Supabase 相关能力只预留接口和字段。
- 配置：支持基础模型配置和云端接入字段，但当前仅本地保存，不做真实调用。

## 三、用户路径

用户进入应用后的标准路径建议如下：

1. 登录应用。
2. 进入项目列表页。
3. 新建项目并输入需求，例如“做一个极简待办清单网页”。
4. 系统调用 LLM 生成网页文件结构。
5. 用户查看生成结果并在编辑页继续追问或微调。
6. 用户在预览页查看真实渲染效果。
7. 用户保存项目，后续可继续编辑或分享。

这个流程的关键不在“生成一次”，而在“连续迭代”。因此编辑页和历史记录设计会比单纯的聊天页更重要。

## 四、技术路线建议（2026 版）

### 1. 现有技术栈存在的问题

原方案里的技术栈方向没错，但有几个明显问题：

- `React + Vite / Next.js` 过于模糊。对当前产品来说是安卓 App 壳 + Web 渲染，不需要同时保留两条前端路线。
- `Zustand / Redux` 也过于宽泛。Redux Toolkit 并不错误，但对当前中小型 AI 应用来说偏重。
- 缺少服务端状态层。只有本地状态管理不够，项目列表、版本记录、用户设置、生成任务状态更适合交给专门的数据获取层。
- 缺少明确版本线。没有指定主版本，后续实施时容易出现用旧版本脚手架的问题。
- 测试与工程化缺失。原方案没有写测试、代码规范、包管理器和 Node 版本，后面会直接影响 CI 和依赖兼容性。
- `Capacitor / React Native WebView` 同时并列，会让团队在第一阶段无法收敛实现路线。

### 2. 推荐最新主栈

结合当前时间点和这个产品的目标，建议直接收敛为以下技术栈：

| 层级 | 推荐技术栈 | 建议版本线 | 说明 |
|---|---|---|---|
| 运行时 | Node.js | 24 LTS | 当前官方 Active LTS，适合新项目 |
| 包管理 | pnpm | 最新稳定版 | 安装更快，适合后续 workspace 扩展 |
| 前端框架 | React | 19 | 当前稳定主线，适合新项目 |
| 构建工具 | Vite | 7.x | 当前主线版本，适合 React 19 项目 |
| 类型系统 | TypeScript | 5.9 | 当前稳定主线，工具链兼容性更好 |
| 路由 | React Router | 7.x | 适合 SPA 和移动端壳应用 |
| 本地状态 | Zustand | 5.x | 轻量，足够支撑编辑器状态和局部 UI 状态 |
| 服务端状态 | TanStack Query | 5.x | 处理数据缓存、请求状态、重试和同步 |
| UI 样式 | Tailwind CSS | 4.x | 当前主线，性能和配置体验更现代 |
| 后端平台 | Supabase | JS Client v2 + Edge Functions | 当前阶段仅预留接入位，后续用于 Auth、DB、Storage、RLS |
| 安卓容器 | Capacitor | 8 | 当前最新主线，已适配更现代 Android 能力 |
| 测试 | Vitest + Playwright | Vitest 4.x | 单测和关键流程测试组合 |
| 规范工具 | ESLint | 9.x | 使用 flat config |

### 3. 2026 版推荐组合

建议主线直接写死为：

- `React 19`
- `Vite 7`
- `TypeScript 5.9`
- `React Router 7`
- `Zustand 5`
- `TanStack Query 5`
- `Tailwind CSS 4.x`
- `@supabase/supabase-js v2（预留）`
- `Supabase Edge Functions（预留）`
- `Capacitor 8`
- `Vitest 4 + Playwright`
- `Node.js 24 LTS + pnpm`

### 4. 为什么这样调整

#### React 19

原方案只写了 React，没有版本线。现在新项目直接上 React 19 更合理，因为它已经是稳定版，而且对 Actions、资源加载和现代并发能力支持更完整。

#### Vite 7，而不是 Next.js

当前产品是安卓 App 包裹的前端应用，核心是本地页面、编辑器、WebView 预览和云同步，不依赖 SSR，也不依赖复杂 SEO，因此 Next.js 在第一阶段并不划算。Vite 7 更轻、更快，也更适合和 Capacitor 配合。

#### Zustand 5 + TanStack Query 5

原方案只写 Zustand 或 Redux，会导致“本地 UI 状态”和“服务端异步状态”混在一起。更合理的拆法是：

- Zustand 管理本地交互状态
- TanStack Query 管理服务端数据状态

这样编辑器状态不会和项目拉取、缓存、重试逻辑耦合在一起。

#### Tailwind CSS 4.x

原方案写 TailwindCSS 方向没问题，但现在不应该再按旧版配置思路来设计。Tailwind 4.x 已经是更现代的主线，配置和性能都比旧版本更适合作为新项目起点。

#### Supabase JS v2 + Edge Functions

原方案里写 Supabase 是对的，但在当前阶段不直接接入真实数据。前端应先按 `@supabase/supabase-js v2` 的未来接入形态设计服务层与类型结构，当前实现只落 mock 和 placeholder，等后续再将模型调用、密钥代理、结果清洗等逻辑接到 Edge Functions 或独立服务层。

#### Capacitor 8

原方案里 `Capacitor / React Native WebView` 并列，现在应该收敛。对于这个产品，Capacitor 8 是更适合的主线，因为它继续强化了现代原生平台适配，尤其是对 Android 端的现代系统栏和平台更新更友好。

### 5. 不推荐继续保留的模糊写法

以下写法建议从方案里完全移除：

- `React + Vite / Next.js`
- `Zustand / Redux`
- `Capacitor / React Native WebView`
- `OpenAI / Anthropic / 自定义 API`

更好的写法应该是：

- 前端框架：`React 19 + Vite 7`
- 状态层：`Zustand 5 + TanStack Query 5`
- 移动封装：`Capacitor 8`
- 模型接入：`统一 AI Adapter + Provider 配置层`

### 6. 最新技术栈下的架构建议

在新技术栈下，推荐按下面的职责拆分：

- React Router 7 负责页面结构
- Zustand 5 负责编辑器局部状态
- TanStack Query 5 负责项目、版本、配置、消息等服务端数据
- Supabase 在后续阶段负责 Auth、Postgres、Storage 和 RLS；当前只预留服务接口与字段
- Edge Functions 负责模型转发、加密和输出校验
- Capacitor 8 负责安卓容器能力
- WebView 负责项目预览

### 7. 技术栈升级后的风险提醒

- React 19、Tailwind 4、ESLint 9 都是现代主线，脚手架和教程容易混入旧写法，实施时要统一模板。
- Capacitor 8 虽然是最新主线，但插件生态是否全部兼容，仍需在立项初期验证。
- Node.js 建议明确统一到 `24 LTS`，否则 Vite、Supabase CLI、ESLint 等工具的支持矩阵容易错位。

### 8. 结论

如果以 2026 年的新项目标准来落地，这份方案的技术栈应该从“方向正确但写法宽泛”升级为“版本明确、职责清晰、能够直接开工”的组合。最核心的调整不是单纯把版本号升高，而是明确这是一套：

`React 19 + Vite 7 + TypeScript 5.9 + Zustand 5 + TanStack Query 5 + Tailwind 4.x + Supabase v2（预留） + Edge Functions（预留） + Capacitor 8`

这样的组合比原方案更现代，也更适合 AI-native 安卓应用的第一阶段落地。

## 五、系统架构

### 1. 架构分层

建议将系统拆为五层：

1. 展示层：登录页、项目列表页、编辑页、预览页、设置页。
2. 业务层：项目管理、会话编排、配置管理、同步逻辑。
3. AI 适配层：统一封装 OpenAI、Anthropic 及自定义兼容接口。
4. 数据层：当前以本地缓存为主，同时预留 Supabase PostgreSQL、对象存储和云端同步接入位。
5. 运行层：Capacitor、WebView、安全沙箱、原生桥接。

### 2. 建议的请求链路

当前阶段请求链路建议为：

`App -> Mock Service Layer -> 返回标准化文件结构 -> 本地缓存 -> WebView 渲染`

未来切换到真实服务时，再升级为：

`App -> API Gateway / Supabase Edge Function -> LLM Provider -> 返回标准化文件结构 -> 本地缓存与云端入库 -> WebView 渲染`

这个链路的重点是“标准化文件结构”，也就是所有模型的返回值最终都要统一为产品内部格式，避免页面逻辑耦合到具体模型厂商。

## 六、核心功能设计

### 1. 登录与用户系统

当前阶段建议同时保留邮箱登录入口和微信登录入口，但都只做 mock 流程与状态展示，不接真实认证。这样可以先把页面、流程和服务接口固定下来，后续再分别替换成邮箱登录与微信桥接登录的真实实现。

用户系统需要具备以下能力：

- 账号注册与登录入口
- 用户资料展示与本地会话
- 多设备同步的后续接入位
- 项目归属隔离的数据模型预留
- 用户级配置管理

### 2. 项目管理

项目管理不应只是一张 `projects` 表，而应该具备完整生命周期：

- 创建项目
- 重命名项目
- 删除项目
- 最近编辑排序
- 项目草稿状态
- 项目版本快照
- 继续对话生成

建议增加“版本快照”能力，否则用户一次错误生成可能直接覆盖全部结果。

### 3. 项目编辑

编辑页是整个产品的核心页面，建议采用三栏或双栏变体布局：

- 输入区：需求输入、追问输入、快捷操作按钮
- 文件区：文件树 + 代码编辑器
- 结果区：生成日志、历史版本、错误提示、操作反馈

移动端空间有限，实际实现可改为分页 Tab 模式：

- `对话`
- `文件`
- `预览`
- `历史`

### 4. 项目预览

项目预览建议采用安全受控的渲染方式：

- 将生成文件写入沙箱目录
- WebView 只访问白名单资源
- 禁止加载任意远程脚本
- 对外链、跳转和危险 API 做限制

如果后续支持多页面项目，还需要一个简单运行时，用于处理页面入口、资源路径和静态文件映射。

### 5. 配置管理

原方案将配置写入 `projects.config`，这一点需要拆分：

- 用户级配置：模型偏好、主题、默认 Base URL、默认 Provider
- 项目级配置：当前项目运行参数、页面主题变量、预览选项

不建议把用户的 API Key 直接明文保存在 `projects.config` 中，否则数据隔离和泄漏风险都比较高。

## 七、LLM 生成机制优化

### 1. 推荐生成模式

不要让模型直接输出“整段自由文本 + 少量代码片段”，而应该强制输出结构化结果。推荐统一返回如下格式：

```json
{
  "projectName": "todo-app",
  "summary": "一个极简待办网页",
  "files": {
    "index.html": "<!doctype html>...</html>",
    "style.css": "body { margin: 0; }",
    "main.js": "const app = document.getElementById('app')"
  },
  "assets": [],
  "messages": [
    {
      "role": "assistant",
      "content": "已生成首页和基础交互逻辑"
    }
  ],
  "meta": {
    "framework": "vanilla",
    "entry": "index.html",
    "version": 1
  }
}
```

### 2. 推荐增加的生成策略

- 首次生成：生成完整最小项目。
- 增量修改：只返回变更文件，降低成本。
- 校验修复：检测 HTML/CSS/JS 错误后再次修复。
- 模板注入：优先从模板生成，再由模型微调。

### 3. 推荐的服务端职责

未来接入真实服务后，服务端或 Edge Function 需要承担以下职责；当前阶段只在服务层中预留这些职责对应的接口：

- 统一鉴权
- 转发模型请求
- 清洗和校验模型输出
- 限流和计费控制
- 敏感词和风险内容过滤
- 失败重试与日志记录

## 八、数据模型设计优化

当前阶段不落真实数据库，但本地 mock 数据结构建议先按未来云端表模型对齐。后续接入真实数据库时，可直接映射到以下核心表：

### 1. 用户表 `profiles`

| 字段 | 类型 | 说明 |
|---|---|---|
| id | uuid | 主键，对应 auth 用户 id |
| nickname | text | 昵称 |
| avatar_url | text | 头像 |
| login_provider | text | 登录来源 |
| created_at | timestamptz | 创建时间 |
| updated_at | timestamptz | 更新时间 |

### 2. 项目表 `projects`

| 字段 | 类型 | 说明 |
|---|---|---|
| id | uuid | 主键 |
| user_id | uuid | 所属用户 |
| name | text | 项目名称 |
| description | text | 项目简介 |
| current_version_id | uuid | 当前版本 |
| status | text | draft / active / archived |
| created_at | timestamptz | 创建时间 |
| updated_at | timestamptz | 更新时间 |

### 3. 项目版本表 `project_versions`

| 字段 | 类型 | 说明 |
|---|---|---|
| id | uuid | 主键 |
| project_id | uuid | 所属项目 |
| version_no | int | 版本号 |
| files | jsonb | 文件快照 |
| summary | text | 本次变更摘要 |
| created_by | uuid | 创建人 |
| created_at | timestamptz | 创建时间 |

### 4. 会话消息表 `project_messages`

| 字段 | 类型 | 说明 |
|---|---|---|
| id | uuid | 主键 |
| project_id | uuid | 所属项目 |
| role | text | user / assistant / system |
| content | text | 消息内容 |
| metadata | jsonb | 模型、耗时、tokens 等 |
| created_at | timestamptz | 创建时间 |

### 5. 用户设置表 `user_settings`

| 字段 | 类型 | 说明 |
|---|---|---|
| user_id | uuid | 主键 |
| theme | text | 主题 |
| preferred_model | text | 默认模型 |
| custom_base_url | text | 自定义接口地址 |
| encrypted_api_key | text | 加密后的密钥 |
| created_at | timestamptz | 创建时间 |
| updated_at | timestamptz | 更新时间 |

### 6. 设计要点

- 用 `project_versions` 做快照，不直接覆盖原始 `files`
- 对话历史独立存储，不塞到 `projects.history`
- 用户设置与项目设置拆开，降低耦合
- 配置表启用 RLS，确保用户数据隔离

## 十、状态管理建议

如果使用 Zustand，建议至少拆成以下几个 store：

- `authStore`：登录态、用户资料
- `projectStore`：项目列表、当前项目、版本信息
- `editorStore`：当前文件、未保存状态、生成中状态
- `settingsStore`：主题、模型、接口配置
- `syncStore`：当前记录本地同步状态，并预留云端同步接入位

建议的数据结构示例如下：

```ts
type ProjectFileMap = Record<string, string>

type ProjectVersion = {
  id: string
  versionNo: number
  files: ProjectFileMap
  summary?: string
  createdAt: string
}

type ProjectState = {
  projects: Project[]
  currentProjectId?: string
  currentVersion?: ProjectVersion
  pendingFiles: ProjectFileMap
  isGenerating: boolean
  isSaving: boolean
  selectProject: (projectId: string) => void
  applyGeneratedFiles: (files: ProjectFileMap) => void
  saveVersion: () => Promise<void>
}
```

这样可以清楚区分“当前已保存版本”和“本地未保存编辑结果”。

## 十一、安卓打包方案建议

### 方案一：React + Capacitor

优点：

- 研发速度快
- 复用 Web 代码最多
- 适合 MVP
- 原生能力接入门槛相对低

缺点：

- 极致性能和复杂原生交互不如纯原生或 React Native

### 方案二：React Native + WebView

优点：

- 更适合复杂原生交互
- 后续扩展空间大

缺点：

- 工程复杂度更高
- Web 和 Native 边界管理更重

### 结论

MVP 阶段推荐优先选择 `React + Capacitor`。当产品验证成功，并且出现大量原生需求时，再评估迁移到 React Native 或混合架构。

## 十二、安全设计与风险控制

### 1. API Key 安全

- 不建议前端明文持久化保存
- 优先通过服务端代理调用模型
- 若支持用户自带 Key，需要加密存储并限制查看

### 2. WebView 安全

- 限制域名访问白名单
- 默认禁止任意下载和外部跳转
- 注入安全策略，限制危险 JS 能力
- 拦截异常请求与错误日志

### 3. 数据隔离

- 当前阶段不启用真实 RLS，但数据模型和服务接口要按用户隔离前提设计
- 后续接入 Supabase 时，所有项目按 `user_id` 做权限隔离
- 后续接入 Supabase 时，用户设置表和项目表都必须有访问策略

### 4. 成本控制

- 增量生成替代全量生成
- 建立 Token 消耗统计
- 对高频用户增加限流
- 可加入模板缓存和结果缓存

## 十三、发布与运维建议

### 1. 环境划分

- `dev`：本地开发环境
- `staging`：联调和测试环境
- `prod`：正式环境

### 2. 日志与监控

需要至少监控以下内容：

- 登录成功率
- 模型调用成功率
- 平均生成耗时
- WebView 渲染失败率
- 项目保存失败率

### 3. 关键指标

产品上线后建议关注：

- 新建项目转化率
- 生成成功率
- 用户二次编辑率
- 项目次日留存
- 平均每用户生成次数

## 十六、最终结论

原始方案方向是对的，尤其是“AI 对话生成网页项目 + App 内预览 + 项目沉淀”的思路很适合做 AI-native 产品。但要真正落地，必须把它从“概念方案”提升为“可执行方案”。本次优化的重点，就是补齐了 MVP 边界、架构分层、数据模型、安全策略和迭代路径。

如果继续往下推进，建议下一步不要直接开始全量开发，而是先把这个方案再拆成一份实施清单，明确第一版只做哪些页面、哪些接口、哪些表和哪些最小功能。这样更容易控制开发节奏，也能更快进入可演示状态。
