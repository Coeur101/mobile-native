## Context

当前登录系统仍使用 `mockAuthService` 和本地假用户持久化，无法提供真实认证、真实会话恢复和真实用户归属。项目文档已经将 Supabase 作为未来云端能力的主路线，因此邮箱登录最合理的第一步是沿 Supabase Auth 落地，而不是再引入新的主认证平台。

当前约束：

- 登录页只有邮箱输入框，没有密码输入和重置密码流程。
- 路由守卫依赖同步读取本地用户状态。
- 项目同时面向 Web 和 Capacitor Android，需要为认证回调保留兼容性。
- 本次目标已经收窄为“先接入邮箱登录”，微信不在范围内。

## Goals / Non-Goals

**Goals:**

- 将邮箱登录从 mock 切换为真实认证。
- 使用与现有路线兼容的免费优先方案落地。
- 引入真实会话持久化和登录恢复。
- 为登录失败、回调处理和环境变量配置建立明确规则。
- 从本次设计中剥离微信登录，避免范围扩散。

**Non-Goals:**

- 本次不实现微信登录。
- 本次不引入新的主认证供应商。
- 本次不实现完整密码体系、重置密码和多因素认证。

## Decisions

### Decision 1: 采用 Supabase Auth 作为邮箱真实认证的主方案

选择：

- 邮箱真实登录使用 Supabase Auth。

原因：

- 现有 PRD 和项目文档已经把 Supabase 作为未来后端主路线。
- 沿现有路线演进，能减少后续账号体系和项目归属的架构漂移。
- Supabase 免费层足以支撑当前 MVP 认证起步。

备选方案：

- Firebase Auth：能力足够，但会偏离现有 Supabase 路线。
- Authing：国内 provider 更丰富，但会让项目过早绑定新平台。

### Decision 2: 第一阶段采用 Email OTP / Magic Link，而不是邮箱密码登录

选择：

- 以邮箱 OTP / magic link 作为首选真实登录模式。

原因：

- 当前登录 UI 只有邮箱输入，OTP / magic link 更贴合现有页面结构。
- 不需要在当前阶段引入密码、密码强度、重置密码等完整密码体系。
- 在免费优先前提下，实现和维护成本更低。

备选方案：

- 邮箱密码登录：可行，但会扩大 UI 与服务范围。

### Decision 3: 为投递稳定性预留自定义 SMTP，而不是把默认测试发信当成正式方案

选择：

- 邮箱认证设计中明确要求使用可配置 SMTP，并优先兼容免费服务商，例如 Resend Free。

原因：

- 真实邮箱登录必须依赖可投递邮件。
- 默认测试发信不应成为正式 MVP 登录路径的长期依赖。

备选方案：

- 平台默认邮件：实现快，但可控性不足。
- 自建 SMTP：维护成本过高。

### Decision 4: 本次变更显式移除微信实现目标

选择：

- 登录页和认证设计中不再把微信当作本次变更交付项。
- 如果 UI 仍保留微信入口，也只能作为未来占位，不得与本次真实认证交付混在一起。

原因：

- 当前最重要的是先拿到一个真实可用的认证入口。
- 微信会显著扩大外部依赖、成本边界和回调复杂度，不适合并入本次邮箱登录 change。

## Risks / Trade-offs

- [Email OTP 依赖邮件可达性] → 通过可配置 SMTP 和真实投递验证降低风险
- [路由守卫当前是同步本地模型] → 在实现中补会话恢复层，避免页面首屏抖动或误跳转
- [暂不做密码登录意味着交互能力有限] → 以 MVP 可用性优先，后续可单独提 change 扩展密码体系
- [UI 中若继续保留微信按钮会造成误解] → 在实现任务中明确要么移除要么标注为未启用
