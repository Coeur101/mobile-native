## Context

当前登录页已经承载了验证码登录、密码登录、注册和找回密码四条路径，但这些路径在一个面板内以近似同权重的按钮和输入区域直接并列，结果是用户进入页面后先看到一组动作，而不是先理解“我现在在哪一步、主操作是什么、次要入口在哪里”。在移动端小屏下，这种排版会进一步放大拥挤感，让切换模式像“乱点按钮”，而不是“沿着流程前进”。

这个 change 的目标不是重写认证能力，而是重写认证入口的交互表达。约束也很清楚：认证后端、7 天登录态、密码与验证码双模式规则保持不变；需要在现有 React + Tailwind 体系内完成；用户可见改动仍需补齐 Playwright UI 验证。

## Goals / Non-Goals

**Goals:**

- 让登录页首屏有明确视觉焦点，而不是一上来出现一组同质按钮。
- 把模式切换设计成“导航选择”或“身份入口选择”，而不是与提交动作混排。
- 为登录、注册、找回密码提供更清晰的渐进披露结构，让每个状态只有一个主 CTA。
- 提升移动端交互感，包括步骤推进、选中态、输入焦点、错误反馈和状态切换过渡。
- 在不修改认证业务规则的前提下，保留现有密码登录、验证码登录、注册、密码重置和 7 天登录态恢复能力。

**Non-Goals:**

- 不新增微信登录或第三方登录入口。
- 不修改 Supabase 认证接口、session TTL 规则或本地持久化模型。
- 不重做整个应用设计系统，只聚焦认证入口页及其直接依赖的轻量 UI 结构。
- 不把营销落地页、帮助中心或完整 onboarding 一并并入本次 change。

## Decisions

### Decision 1: 认证首页采用“入口层 + 过程层”的双层结构

选择：

- 首屏先展示品牌说明、价值感知和当前推荐入口。
- 第二层再承载当前选中的认证流程表单。
- 模式切换区域单独作为“入口导航”，不与提交按钮同组。

原因：

- 用户当前最强烈的不满是所有动作被做成一排按钮，缺少交互层级。
- 双层结构可以把“先选路径”和“再执行动作”拆开，降低认知冲突。
- 这比在同一块卡片里继续堆按钮更容易形成移动端节奏。

备选方案：

- 保持单卡片结构，仅替换按钮样式：视觉会改善，但结构问题还在。
- 改成多页面跳转：路径更清晰，但切页成本更高，不利于认证状态回流。

### Decision 2: 模式切换使用分段导航或卡片入口，不再直接使用等权按钮矩阵

选择：

- 验证码登录、密码登录、注册、找回密码通过顶部 segmented control、横向选项卡或卡片式入口承载。
- 当前选中的模式要有强选中态，但不触发提交。

原因：

- 切换模式本质是导航，而不是提交动作。
- 把模式切换与 CTA 解耦之后，用户才能一眼知道哪个是“切换”，哪个是“继续”。

备选方案：

- 下拉选择模式：节省空间，但可发现性差。
- 继续用四宫格按钮：实现最省，但正是本次要解决的问题。

### Decision 3: 每个认证状态只保留一个主 CTA，并把辅助动作下沉

选择：

- 每个状态只保留一个高权重主按钮。
- 次要动作如“切换登录方式”“返回登录”“忘记密码”使用文本按钮、次级链接或底部辅助区承载。
- 说明性文案与异常反馈与当前步骤绑定，不跨模式共享一个混杂区域。

原因：

- 移动端最忌多个主按钮同屏争抢注意力。
- 主 CTA 唯一化后，交互才会产生明确推进感。

备选方案：

- 每个区域保留两个按钮：视觉仍会竞争。
- 把全部辅助动作隐藏到菜单：会降低可发现性。

### Decision 4: 引入明显的步骤反馈和页面状态动画，但保持实现轻量

选择：

- 利用现有 `motion/react` 做小范围过渡，如模式内容切换、卡片展开、错误提示进入、输入区焦点强调。
- 不增加新的动画库，不做复杂手势系统。

原因：

- 用户反馈里“没有交互的感觉”不仅是排版问题，也包含状态切换缺乏反馈。
- 轻量过渡足以提升“我在操作一个流程”的感受，同时不会把 change 扩大成全面动效重构。

备选方案：

- 完全静态切换：实现简单，但难以解决“没交互感”。
- 引入复杂动画框架：收益不成比例。

### Decision 5: 保持认证服务与业务状态机不变，UI 只消费既有能力

选择：

- `authService`、`useAuthStore`、TTL 恢复、密码重置回流等能力继续沿用。
- 本次主要重构 `LoginPage` 的布局、分层、文案与交互组织。

原因：

- 当前问题集中在界面组织，而不是认证后端不可用。
- 保持业务层稳定，可以让这次 change 聚焦在真正被用户指出的问题上。

备选方案：

- 顺手调整认证状态机：会扩大风险，也偏离核心诉求。

## Risks / Trade-offs

- [视觉层级提升后，代码结构会比现在更复杂] → 通过拆分局部渲染块或小组件控制复杂度，而不是把所有状态继续堆在一个函数里。
- [过多动画可能影响移动端性能] → 只保留对状态理解有帮助的过渡，不做大面积持续动画。
- [入口层和过程层拆开后，文案可能变多] → 用更短的标题、提示和辅助文本控制密度。
- [UI 改动不触碰业务层，可能掩盖部分现有交互瑕疵] → 用 Playwright 覆盖主流程，确保重构后没有损伤已有认证能力。
