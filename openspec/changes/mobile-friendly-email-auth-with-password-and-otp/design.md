## Context

项目刚完成真实邮箱 OTP 登录接入，但当前实现仍以单一路径为主：用户输入邮箱，接收验证码或 magic link，完成登录。这个模型虽然能替代 mock auth，但还不适合作为移动端长期使用的账号体系，因为它缺少注册时的密码建立、后续密码登录、密码重置边界，以及显式的“一周记住我”策略。

当前约束与背景：

- 前端栈为 React 19 + TypeScript + Vite 7，移动端通过 Capacitor Android 封装。
- 当前认证服务已基于 Supabase Auth，登录页和路由守卫已经接入真实会话恢复。
- 当前项目数据仍主要依赖本地 mock 持久化，`profiles`、`projects`、`project_versions`、`project_messages`、`user_settings` 仅在文档层定义，还没有在本 change 语义下被纳入配置边界。
- 用户明确偏好移动端友好体验，不接受“每次登录都同时输入密码和验证码”的重流程。
- 项目治理要求所有用户可见改动都补齐 Playwright UI 自动验证，且测试生成物必须在记录结果后删除。
- 本 change 仍需遵守三阶段工作流、`design_review` / `implementation_review` 双 review、`TASK.json` 登记与 `logs/<task-id>.jsonl` 追加式历史要求。

## Goals / Non-Goals

**Goals:**

- 将邮箱认证升级为移动端优先的注册与登录体系。
- 支持首次注册时“邮箱验证码 + 设置密码”的分步流程。
- 支持后续登录时“验证码登录”与“密码登录”双模式共存。
- 为高风险操作预留追加验证码校验的状态机挂点，但不将其作为本次默认日常登录流程。
- 将客户端登录态保留策略显式定义为 7 天，并在本地恢复时校验 TTL。
- 为当前用户表和项目表建立明确配置边界，保证认证用户与项目归属关系可以向真实后端迁移。
- 扩展错误提示、密码重置入口、会话恢复与退出登录清理逻辑。
- 让任务拆分、验证证据、临时测试清理与项目工作流保持一致。

**Non-Goals:**

- 本次不实现微信登录。
- 本次不做“每次登录强制密码 + 验证码双输”的完整双因子体验。
- 本次不实现设备指纹、风控评分或后台安全审计平台。
- 本次不改变 `development-workflow` 规范本身，只在该规范下交付本功能。
- 本次不实现完整项目云同步或多人协作。

## Decisions

### Decision 1: 采用“注册双步骤、登录双模式”而不是“日常登录强制双因子”

选择：

- 注册：邮箱 -> 发送验证码 -> 验证邮箱 -> 设置密码 -> 完成注册。
- 登录：允许用户选择“验证码登录”或“密码登录”。

原因：

- 这更符合手机端输入成本低、单手操作简单的目标。
- 用户已经明确要求移动端方便操作，因此不应把完整强制双因子塞进默认登录主路径。
- 该策略同时满足“支持密码”和“支持验证码”的诉求，也为未来高风险场景的验证码追加校验保留空间。

备选方案：

- 每次登录都要求密码 + 验证码：安全性更强，但移动端体验过重。
- 仅保留 OTP：实现最轻，但无法满足密码登录与密码找回需求。

### Decision 2: 继续基于 Supabase Auth，同时引入 password 与 OTP 并行状态机

选择：

- 继续使用 Supabase Auth 作为唯一认证后端。
- 注册时使用 email/password 账号创建 + 邮箱验证。
- 登录时同时支持 `signInWithPassword` 与 `signInWithOtp` / `verifyOtp`。

原因：

- 当前项目已经完成 Supabase 接入，继续沿用可避免技术栈再次漂移。
- Supabase 已能覆盖密码登录、邮箱 OTP、密码重置和会话恢复这些本次所需能力。
- 使用同一后端可保持 session、用户元数据、TTL 策略和密码重置流程的一致性。

备选方案：

- 引入新的认证供应商：会增加额外迁移和维护成本。
- 自建密码与验证码服务：超出当前项目复杂度边界。

### Decision 3: 认证 UI 采用单页分步流，优先照顾手机端输入负担

选择：

- 登录页或认证页保持单容器分步式交互。
- 第一层只暴露邮箱输入与主要 CTA。
- 根据状态展开验证码输入、密码输入、设置密码、忘记密码入口。

原因：

- 手机端更适合渐进披露，而不是一次堆满邮箱、密码、确认密码、验证码、重置入口。
- 分步流更容易在未来加入高风险验证码校验，而不把界面挤爆。

备选方案：

- 多 Tab 登录/注册页：可行，但在小屏上容易引起状态切换混乱。
- 单页全字段表单：实现简单，但移动端操作负担明显更高。

### Decision 4: 客户端登录态显式增加 7 天 TTL，而不是无限期依赖本地快照

选择：

- 在持久化 auth state 中加入如 `rememberedAt` / `rememberUntil` 的显式时间戳。
- 应用启动时先检查 TTL，再决定是否尝试恢复 Supabase session。

原因：

- 用户要求“登录信息保存在客户端一个星期”，这必须是明确规则，而不是隐含依赖 session 是否还在。
- 这能让产品行为更稳定，也让测试能覆盖“未过期恢复”“已过期强制清理”两类场景。

备选方案：

- 无限期保留本地状态：体验方便，但不满足需求。
- 每次都只依赖 Supabase 默认会话生命周期：规则不透明，难以对齐产品预期。

### Decision 5: 使用 task 级闭环交付本 change

选择：

- 以认证基础、注册/登录流程、TTL 持久化、验证与文档为原子 task 拆分。
- 每个 task 在 `TASK.json` 中登记质量门禁与 `historyRef`。
- 每个 task 的测试结果、构建结果、UI 自动化结果写入 `openspec/changes/<change>/logs/<task-id>.jsonl`。
- 临时 Vitest/Playwright 用例与 `playwright-report`、`test-results`、截图、trace 等在记录结果后立即删除。

原因：

- 这与项目已有 `task-level-closed-loop-workflow` 一致。
- 认证改造涉及 UI、状态机、会话和外部系统，拆 task 可避免“一个大任务一次做完无法回溯”。

### Decision 6: 将用户表与项目表配置纳入同一 change，而不是留到后续再补

选择：

- 在本 change 中同时定义 `profiles`、`projects`、`project_versions`、`project_messages`、`user_settings` 的配置边界。
- 以“当前用户身份”和“项目归属关系”作为认证落地后的紧邻一层，而不是继续停留在纯本地 mock 数据。

原因：

- 一旦登录从 mock 变成真实账号，用户数据和项目数据如果没有归属配置，就会形成“认证是真的，数据还是匿名本地”的断层。
- 用户表和项目表的主键、外键与归属关系会直接影响认证后的 session 恢复、项目列表加载和后续数据迁移。
- 把这层放进本 change，可以在设计阶段就定清数据边界，减少后续返工。

备选方案：

- 另开一个只做数据表配置的 change：拆分更细，但会导致本次认证 change 对“登录后数据属于谁”没有完整答案。

## Risks / Trade-offs

- [认证状态机显著变复杂] → 通过“注册、验证码登录、密码登录、TTL 恢复”分层拆 task，并在 spec 中为每条路径定义场景。
- [Supabase 密码流与 OTP 流并存，容易出现 UI 混淆] → 采用分步式移动端交互，而不是把所有字段同屏暴露。
- [7 天 TTL 与服务端 session 生命周期可能不完全一致] → 本地 TTL 仅作为客户端恢复门槛，真实 session 仍以 Supabase 返回结果为准。
- [认证先落地但用户/项目表未配置，会出现数据归属断层] → 在本次 change 中同步定义用户表、项目表及其关联结构。
- [高风险二次验证如果现在就强推，会拉高移动端摩擦] → 本次只预留状态机挂点，不默认纳入每次登录。
- [测试与临时产物增多] → 继续使用任务级验证和删除策略，先记结果再删文件，避免仓库负担累积。

## Migration Plan

1. 在 Definition 阶段完成 proposal、design、spec、tasks，并通过 `design_review`。
2. 在 Delivery 阶段先重构 auth state 与持久化模型，再接入注册/登录双模式和 7 天 TTL。
3. 为密码登录、验证码登录、注册验证、过期恢复分别补功能测试和 Playwright UI 测试。
4. 将临时测试产物摘要写入任务历史后删除生成物。
5. 在 Closure 阶段同步 `PROJECT.md`、`docs/PRD.md`、`CHANGELOG.md`、接入说明，完成 `implementation_review`、commit 与 archive。

回滚策略：

- 若密码登录链路不稳定，可回滚到当前已完成的 OTP-only change。
- 若 7 天 TTL 引起异常登出，可先退回“仅依赖真实 session 恢复”的策略，但需要明确修改 spec。

## Open Questions

- 高风险操作的验证码追加校验是只预留接口，还是本次就先在某个具体入口落一个场景？
- 注册成功后是否默认直接登录，还是要求用户先显式完成邮箱验证后再进入应用？
- “一个星期”是否按自然 7*24 小时精确计算，还是允许“7 天内最近一次活跃续期”策略？当前倾向前者。
