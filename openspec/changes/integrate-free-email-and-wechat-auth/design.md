## Context

当前登录能力停留在 mock 阶段：邮箱登录只保存一个本地假用户，微信登录按钮会直接模拟成功。这种状态已经不满足真实会话、用户隔离和后续云端项目持久化的前提。

项目现有约束：

- 代码和 PRD 都已经预留了 `Supabase` 作为未来云端能力接入位。
- 登录页当前只有邮箱输入框，没有密码输入、忘记密码、重设密码等完整密码体系 UI。
- 项目是 React + Vite + Capacitor Android，认证回调需要兼容 Web 与移动壳。
- 用户明确要求优先使用免费方案。

外部边界：

- Supabase Auth 官方支持邮箱认证和多个 social providers，但不包含 WeChat 原生 provider。
- 微信真实登录依赖微信开放平台资质，不存在稳定的零成本生产方案。

## Goals / Non-Goals

**Goals:**

- 用与现有架构一致的免费方案落地真实邮箱登录。
- 让登录态从本地 mock 迁移到真实会话。
- 把微信登录从“mock 成功”改成“真实接入路径 + 能力门控”。
- 明确当微信资质或凭据不存在时，产品和代码应如何表现。
- 为后续实现补齐环境变量、错误态、回调和验证任务。

**Non-Goals:**

- 本变更不承诺交付零成本的生产级微信登录。
- 本变更不切换整个项目到新的主认证供应商。
- 本变更不覆盖账号合并、多设备会话管理、企业级风控。

## Decisions

### Decision 1: 邮箱真实登录采用 Supabase Auth，优先选择 Email OTP / Magic Link

选择：

- 邮箱登录使用 Supabase Auth。
- 第一阶段采用 Email OTP / Magic Link，而不是邮箱密码登录。

原因：

- 项目文档已经把 Supabase 作为未来后端主路线，沿用它可以减少架构漂移。
- 当前登录页只有邮箱输入，没有密码体系；OTP / magic link 更贴合现有 UI 和 MVP 复杂度。
- 免费方案下，邮箱 OTP 的落地成本低于完整密码体系。

备选方案：

- 邮箱密码登录：可行，但需要新增密码、重置密码、密码策略和更多错误处理，当前收益不高。
- Firebase Auth：可行，但与项目既有 Supabase 方向不一致。
- Authing：支持更多国内 provider，但会把项目主认证依赖切到另一个平台。

### Decision 2: 邮件发送采用自定义 SMTP 免费额度，而不是依赖默认测试配置

选择：

- 为 Supabase Auth 配置自定义 SMTP。
- 首选具有免费额度的邮件服务，例如 Resend Free。

原因：

- 真实邮箱登录必须依赖稳定可投递邮件。
- 默认测试发信能力不适合作为正式 MVP 登录链路。
- 自定义 SMTP 能让免费邮箱登录真正可验证。

备选方案：

- 使用平台默认邮件：实现快，但可控性和投递稳定性不足。
- 自建 SMTP：维护成本高，不适合当前阶段。

### Decision 3: 微信登录采用“能力门控 + 真实接入位”，不再允许 mock 成功

选择：

- 登录页继续保留微信入口，但服务层必须先判断配置与能力是否满足。
- 当缺少微信开放平台资质、AppID、Secret、回调地址或 broker 配置时，微信登录必须进入明确受限状态，而不是模拟成功。
- 只有在真实凭据齐备时，才允许走真实微信 OAuth 流程。

原因：

- 当前最严重的问题不是“还没接微信”，而是“微信按钮会伪造成功”，这会误导后续认证设计。
- 零成本生产级微信登录不现实，必须把成本边界变成系统行为。
- 能力门控可以让项目先完成邮箱真实认证，同时为未来微信接入留出稳定接口。

备选方案：

- 继续保留 mock 成功：被拒绝，原因是会破坏真实认证语义。
- 直接强行接第三方聚合认证：成本和架构漂移都过大。

### Decision 4: 微信真实登录通过独立 broker 层接入，而不是硬塞到前端直接完成

选择：

- 为微信登录设计独立 broker / callback 接口层。
- 前端只负责发起登录、处理回调结果和显示能力状态。

原因：

- WeChat 不在 Supabase 原生 provider 列表内，直接前端式 social login 不成立。
- broker 层更适合处理 code 交换、签名校验、openId / unionId 映射和后续会话桥接。
- 即便当前阶段不落地完整 broker，也应该先把接口边界设计出来。

备选方案：

- 前端直接拼 OAuth 流程：安全性和可维护性都差。
- 改用另一个全家桶认证供应商：代价过高。

## Risks / Trade-offs

- [微信生产接入并非真正免费] → 在产品和实现里明确能力门控，避免承诺不存在的零成本交付。
- [Email OTP 依赖发信可达性] → 引入自定义 SMTP，并在任务中加入真实投递验证。
- [Supabase 与微信桥接存在实现复杂度] → 先把 broker 作为独立接口边界写清，再在实现阶段逐步落地。
- [Capacitor 回调处理复杂] → 在设计中提前约束 redirect URI、deep link 和 Web fallback。
