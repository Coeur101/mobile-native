## Context

当前项目已经接入 Supabase 邮箱 OTP，并在移动端登录页中同时保留了验证码登录、密码登录、注册设密码和密码重置四条路径。这个模型虽然可用，但已经和新的产品方向冲突：登录验证要统一为邮箱验证码，密码不再作为日常登录入口，而是变成登录后可选开启的安全增强能力。

与此同时，当前用户资料仍然是由认证会话快照和本地持久化共同拼出来的轻量对象，只包含邮箱、昵称和认证时间，没有正式的 Supabase profile 记录，也没有头像、密码配置状态等账户资料边界。设置页当前仍承担应用设置中心角色，而不是用户账户中心。

这次 change 同时触及认证模型、用户资料模型、主页入口、个人信息页和本地缓存边界，属于跨模块重构，并且涉及 Supabase 数据模型与 App 端交互一致性，因此需要先明确设计决策再进入实现。

## Goals / Non-Goals

**Goals:**

- 将登录入口统一为邮箱验证码认证，移除登录页上的密码登录入口。
- 将首次注册改为“验证码验证成功后直接创建账户并生成默认资料”，不再要求首次注册立即设置密码。
- 引入 Supabase 用户资料记录，作为昵称、头像、密码状态等用户信息的权威来源。
- 将主页右上角入口从“设置”替换为“用户信息”，个人信息页展示头像、昵称、邮箱、主题切换与安全设置。
- 允许用户上传头像，客户端先压缩，再转成 base64 后写入 Supabase 资料记录。
- 将密码改为可选安全增强能力；设置密码或重置密码前都必须先通过邮箱验证码确认。
- 收缩原设置页范围，仅保留主题切换相关能力，其余演示型设置从主入口移除。

**Non-Goals:**

- 不在本次 change 中引入微信登录或任何第三方登录。
- 不在本次 change 中引入对象存储、公开文件 URL、图片 CDN 或路径型头像存储。
- 不在本次 change 中重做项目、版本、消息等业务表的完整云端持久化实现，只定义与账户资料相关的边界。
- 不在本次 change 中新增复杂多因子认证体系；邮箱验证码仅用于登录和密码安全操作的确认。

## Decisions

### Decision 1: 认证模型改为 OTP-first，密码从登录主路径中退出

选择：

- 登录页只保留邮箱验证码登录主路径。
- 注册流程在邮箱验证码验证成功后直接完成账户创建并进入应用。
- 密码不再作为登录入口暴露，而是改为登录后在个人信息页中手动设置的安全增强能力。

原因：

- 这最符合“所有登录验证统一使用验证码输入方式、兼容 App 端”的目标。
- 统一验证心智模型后，登录入口可以明显简化，避免继续维持 OTP / password 双轨切换。
- 密码如果继续作为登录路径存在，UI 和 spec 最终仍会回到双模型并存。

备选方案：

- 保留底层密码登录能力但 UI 不展示：迁移更平滑，但会让规格和代码长期不一致。
- 继续并存 OTP / password：与用户已经明确确认的新方向相冲突。

### Decision 2: 用户资料以 Supabase profile 记录为准，本地只缓存会话与少量快照

选择：

- 新增 Supabase-backed 用户资料记录，承载昵称、头像 base64、密码配置状态、更新时间等字段。
- 本地仍允许保留认证会话和必要的恢复窗口，但用户资料不再以 `localStorage` 为权威来源。

原因：

- 用户已经明确要求“用户信息不应该存储在本地，而应存储在 Supabase 下”。
- 当前 `localDb` 只适合作为会话缓存，不适合作为正式账户资料库。
- 资料云端化后，App 端和未来多端访问才可能保持一致。

备选方案：

- 完全不在本地缓存资料：一致性最好，但每次首屏都依赖拉取资料，移动端体验更脆弱。
- 继续以本地 profile 为主：与新的账户边界相违背。

### Decision 3: 头像采用“客户端压缩 + base64 存 Supabase profile”策略

选择：

- 用户选择头像后，客户端先执行尺寸压缩与质量压缩，再转为 base64 存入 profile 记录。
- 不引入路径、对象存储或上传后返回 URL 的方案。

原因：

- 当前目标是减少基础设施成本和实现复杂度。
- 项目现阶段还没有正式的媒体存储体系，base64 是最短链路方案。
- 只要压缩策略足够严格，头像字段可以控制在可接受大小内。

备选方案：

- Supabase Storage + 路径/URL：长期更合理，但会额外引入 bucket、上传权限、清理和访问策略。
- 直接存原始 base64：实现最省，但体积不可控，后续会明显反噬 profile 拉取成本。

### Decision 4: 个人信息页接管原设置主入口，主题切换内聚到个人信息页

选择：

- 首页右上角从“设置”按钮改为“用户入口”按钮。
- 个人信息页展示头像、昵称、邮箱、主题切换和安全设置。
- 原设置页功能缩减为主题相关能力，或者被个人信息页吸收。

原因：

- 当前设置页里的 Base URL、API Key、备注等能力更像临时演示配置，而非稳定产品能力。
- 用户入口应该优先服务“我是谁”和“我的账户”，而不是开发期调试项。
- 个人信息页成为统一入口后，密码安全能力和资料维护就有自然落点。

备选方案：

- 保留独立设置页，再新增个人信息页：信息架构会重复，入口也会变多。
- 继续把用户资料混在设置页里：语义不清晰。

### Decision 5: 密码设置与重置都走“验证码确认后再设新密码”的页内流程

选择：

- 未设置密码时显示“设置密码”。
- 已设置密码时显示“重置密码”。
- 两种操作都要先向邮箱发送验证码，完成验证码确认后，才允许输入新密码。

原因：

- 这样可以保持整个账户体系只有一种验证方式，即邮箱验证码。
- 避免在 App 端继续保留 magic link 回流和页内 OTP 两套模型。
- 也符合“密码是可选安全增强能力，而不是主认证路径”的产品定义。

备选方案：

- 继续用 `resetPasswordForEmail` 邮件回流链接：可复用现有 Supabase 能力，但会让验证模型重新分裂。
- 设置密码不做邮箱确认：安全性不足，与用户要求不符。

## Risks / Trade-offs

- [头像存为 base64 会让 profile 记录变重] → 强制客户端压缩尺寸和质量，并限制头像上传规格。
- [Supabase profile 成为新权威后，本地旧 profile 缓存可能产生脏数据] → 在迁移时增加本地资料字段清理和优先级切换逻辑。
- [把密码移出登录页会影响已有密码登录用户心智] → 在个人信息页提供清晰的“密码仅用于账户安全增强”说明，并在过渡期妥善处理已配置密码的账户。
- [原设置页缩减后，部分演示配置入口会消失] → 明确将这些开发期字段移出主用户入口，必要时再另行定义开发者设置入口。
- [若直接完全移除密码登录 spec，现有实现与新规范之间会有迁移窗口] → 通过新的 change 任务拆分，先完成 spec 和 UI 收敛，再逐步清理旧接口与旧状态字段。

## Migration Plan

- 第一步：补充 OpenSpec delta specs，明确 OTP-only 登录、云端 profile、个人信息页和密码安全增强能力的新要求。
- 第二步：在 Supabase 侧补齐 profile 记录边界与默认资料创建逻辑，并明确 `has_password` 等字段语义。
- 第三步：重构登录页，去掉密码登录入口和注册后立即设密码路径，统一到 OTP 登录 / OTP 注册。
- 第四步：新增个人信息页并改造主页入口，同时把主题切换迁入个人信息页。
- 第五步：实现头像压缩 + base64 持久化，以及密码设置/重置前的验证码确认流程。
- 第六步：清理不再使用的本地资料持久化路径和演示型设置字段。
- 回滚策略：在迁移窗口内保留现有 Supabase auth 核心会话能力；如果 profile 云端化流程失败，可以先回退到旧登录页和旧资料读取逻辑，但不应部分提交 specs。

## Open Questions

- 个人信息页最终是否完全取代 `/settings` 路由，还是保留 `/settings` 作为兼容别名跳转到个人信息页。
- 头像压缩的具体尺寸、格式和大小阈值是否要在 spec 中写成强约束，还是先以实现约定落地。
- `has_password` 是通过 profile 字段显式维护，还是通过认证侧能力推断；两者对一致性和实现复杂度影响不同。
